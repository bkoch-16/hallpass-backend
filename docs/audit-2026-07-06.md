# Codebase Audit — 2026-07-06

**Scope:** all three APIs (`user-api`, `schools-api`, `passes-api`) plus shared packages (`auth`, `db`, `logger`, `types`), focused on business logic and high-level design patterns. Cloud-readiness of passes-api (new, untested on Cloud Run) reviewed specifically.

**Method:** full read of every source file in `apps/*/src` and `packages/*/src`, the Prisma schema, seed, and deploy workflow. Items already tracked in `tech-debt.md` are referenced, not duplicated.

**Overall:** the codebase is in good shape. The pass lifecycle (status-guarded `updateMany` transitions, atomic Lua slot claims with rollback, idempotent expiry jobs, reconcile endpoint) is carefully built and the tricky races are handled. The findings below are mostly product-level business-logic gaps and cross-service design drift — not implementation bugs in the state machine.

---

## Critical

### C1. passes-api will not process expiry jobs reliably on default Cloud Run settings
**Status 2026-07-06: dev environment set up and verified; accepted tradeoff for dev; still fully open for prod (see resolution below).**

`apps/passes-api/src/index.ts:25` starts the BullMQ worker in-process, and `apps/passes-api/src/lib/queue.ts:119` runs it on the same instance that serves HTTP. On Cloud Run's default request-based billing, CPU is throttled to near zero between requests and the service can scale to zero — so delayed expiry jobs sit in Upstash and only get processed when a request happens to wake an instance. Pass auto-expiry timing is a core business behavior.

The deploy workflow (`.github/workflows/deploy.yml`) passes no service flags — everything is managed out-of-band on the Cloud Run service.

**Dev resolution (done 2026-07-06):** `passes-api-dev` was configured and is healthy. Cost decision: **no** `min-instances`/`--no-cpu-throttling` on dev — scale-to-zero is accepted, which makes expiry timing best-effort. The Cloud Scheduler job `passes-reconcile-expiry-dev` (every 10 min, bearer `INTERNAL_SECRET_DEV`) doubles as recovery *and* heartbeat, so worst-case expiry lag on dev is ~10 minutes. Session affinity is enabled. Verified end-to-end: `/health` 200, `POST /internal/reconcile-expiry` → `{"scheduled":0,"reconciled":3}`.

**Prod setup — Phase A done 2026-07-06:**
- Secrets created: `REDIS_URL` (same value as dev — **shared Upstash DB decision**, see below) and `INTERNAL_SECRET` (fresh, distinct from dev), both with `secretmanager.secretAccessor` for the compute SA.
- Prod DB migrations applied (`add_pass_model`, `add_pass_requester`, `add_denier_note`); verified `one_active_pass_per_student` CREATE INDEX is present in the migration with no DROP (M5 check).
- **Shared Redis decision (2026-07-06):** Upstash free tier allows one database, so dev and prod share it. All passes-api Redis keys (BullMQ, slot counters, socket.io adapter channel) are now namespaced by a required-per-env `REDIS_PREFIX` env var (branch `setup-passes`); without it a dev worker would consume prod expiry jobs. `passes-api-dev` already has `REDIS_PREFIX=dev` set.
- **Phase B (after `setup-passes` → develop → main):** the first main deploy creates the `passes-api` service (first revision fails on missing env — expected). Then: `gcloud run services update passes-api` with the five prod secrets + `CORS_ORIGIN` + `REDIS_PREFIX=prod` + `--session-affinity`; grant `allUsers` run.invoker; create scheduler job `passes-reconcile-expiry` (`*/10`, bearer prod `INTERNAL_SECRET`). Full commands in `docs/INFRA.md`.
- **Cost decision (2026-07-06): no min-instances / CPU-always-allocated on prod either** — this is a side project and scale-to-zero is the deliberate choice. Consequence: pass expiry fires on time only while traffic keeps an instance warm; otherwise it lags up to the scheduler interval. The reconcile scheduler job is therefore **mandatory** infrastructure on prod, not optional recovery, and its interval is the worst-case expiry lag (run it as frequently as tolerable). Revisit only if real users complain about stale passes.
- ~~**Upstash eviction policy:** boot logs showed BullMQ's warning `Eviction policy is optimistic-volatile. It should be "noeviction"` — under memory pressure Upstash could evict delayed jobs.~~ **Done 2026-07-06:** eviction set to `noeviction` in the Upstash console.

**Lessons captured for the runbook (belongs in `docs/INFRA.md`):** a brand-new Cloud Run service created by the deploy workflow starts with *zero* env config and *no* public invoker access, because the workflow intentionally passes none — first deploys always fail until someone runs the one-time setup: create `*_DEV`/prod secrets (grant `secretmanager.secretAccessor` to both the compute and github-actions service accounts), `gcloud run services update --set-secrets=... --set-env-vars=CORS_ORIGIN=...`, and `gcloud run services add-iam-policy-binding --member=allUsers --role=roles/run.invoker`. This is exactly how the 2026-07-06 `passes-api-dev` first deploy failed (revision 00001, "container failed to listen on PORT=8080" — actually a ZodError from `env.ts` on missing vars).

### C2. Per-IP, in-memory rate limits will lock out an entire school
All three apps use `express-rate-limit` with the default per-IP key and in-memory store: 100 req/15 min general (`apps/user-api/src/app.ts:44`, `apps/schools-api/src/app.ts:48`, `apps/passes-api/src/app.ts:39`) and 10 req/15 min for auth (`apps/user-api/src/app.ts:52`).

A school's students all sit behind one NAT IP. That's **100 total API requests and 10 total logins per 15 minutes for an entire campus** — the product breaks on day one of a real deployment (first period: hundreds of students opening the app and logging in from the same IP). Separately, the in-memory store means each Cloud Run instance keeps its own counters, so limits are inconsistent and reset on scale events.

**Fix direction:** key authenticated traffic by user id (after `requireAuth`) instead of IP, raise the per-IP auth limit substantially or key it per-email, and back the store with Redis on passes-api (it already has Redis) or accept per-instance counters knowingly. This belongs in the shared middleware package (see recommendation below) so all three services get the same policy.

## Major

### M1. Admin-provisioned users can never log in
`POST /api/users` and `POST /api/users/bulk` (`apps/user-api/src/routes/user.ts:120`, `:154`) create `User` rows directly via Prisma — no better-auth credential `Account`, no password, no invite. These users cannot sign in, and better-auth's email/password sign-up will refuse their email because the `User` row already exists. The seed (`packages/db/prisma/seed.ts:15-25`) works around this by hand-crafting `Account` rows with a scrypt hash that must "match better-auth's password hashing config exactly" — itself a version-upgrade landmine, and not something the API does.

`docs/SCHEMA_PLAN.md` documents `/users/bulk` as "bulk create for school onboarding" but defines no onboarding/claim flow. **The primary onboarding path produces accounts that cannot authenticate.** Decide on a flow (invite token + set-password endpoint, or better-auth admin plugin to create credentialed users) before any real school is onboarded.

### M2. `PATCH /api/users/:id` is missing the target-rank check that DELETE has
DELETE refuses to act on a target whose rank ≥ the caller's (`apps/user-api/src/routes/user.ts:277`). PATCH has no equivalent (`user.ts:194-249`): it only checks the *assigned* role isn't above the caller's rank (`user.ts:230`) and that email changes come from ADMIN+ (`user.ts:225`). Consequences for a school ADMIN:
- can edit a **peer ADMIN**'s email — i.e. redirect another admin's login identity (account-takeover adjacent);
- can demote a peer ADMIN to STUDENT;
- can promote a STUDENT to ADMIN (equal rank passes the `>` check — if peer creation is intended, peer *modification* probably still isn't).

Mirror the DELETE rule: non-self targets must be strictly below the caller's rank.

### M3. School timezone is unvalidated free text and failure modes are inconsistent
`createSchoolSchema`/`updateSchoolSchema` accept any string for `timezone` (`apps/schools-api/src/schemas/school.ts:18`, `:25`). In passes-api, `getTodayInTimezone`/`getCurrentTimeInTimezone` silently fall back to UTC on a bad zone (`apps/passes-api/src/lib/time.ts:1-24`), but `tzOffsetMs`/`localMidnightAsUTC` have no guard (`time.ts:26-52`) and throw `RangeError` — so a bad timezone makes pass creation 500 *after* the row is inserted (via `periodEndDate` at `apps/passes-api/src/routes/passes.ts:283-291`) and makes quota checks throw (`getIntervalStart`). Validate with `Intl.supportedValuesOf("timeZone")` (or a try/`DateTimeFormat` probe) in the schema and the whole family of failure modes disappears.

### M4. Bulk calendar upsert is non-atomic and reports nothing on partial failure
`POST /schools/:schoolId/calendar` loops entries and returns a bare 422 mid-loop when a later entry references a bad schedule type (`apps/schools-api/src/routes/calendar.ts:66-79`) — earlier entries are already written and the client can't tell. Contrast with `/users/bulk`, which returns per-item results. Either wrap the loop in `prisma.$transaction`, or validate all entries up front, or adopt the per-item result shape. (Also an N+1: one `findUnique` + one write per entry.)

### M5. The partial unique index landmine (documented but unguarded)
`one_active_pass_per_student` exists only in a migration; the schema comment (`packages/db/prisma/schema.prisma:188-197`) warns every future `prisma migrate dev` will generate a `DROP INDEX` that must be hand-deleted. passes-api's 409 duplicate-pass contract (`apps/passes-api/src/routes/passes.ts:69-73`, `:229`) silently degrades if it's ever applied. A comment is not a guard — add a CI check that greps generated migrations for `one_active_pass_per_student` (or a post-migrate assertion that the index exists) so the failure is loud.

### M6. Schools-api mutations ignore in-flight pass state that passes-api depends on
Cross-service invariants have no owner:
- **Deleting a destination** doesn't check for in-flight passes, and `promoteFromQueue` doesn't filter deleted destinations (`apps/passes-api/src/lib/slots.ts:196-201` includes only `maxOccupancy`) — WAITING students get promoted into a destination that no longer exists. Note `scheduleType` DELETE *does* guard its references (`apps/schools-api/src/routes/scheduleType.ts:115-122`); destination DELETE (`destination.ts:94`) has no equivalent.
- **Shrinking `maxOccupancy`** leaves the live Redis counter at the old capacity until the scheduled reconcile.
- **Changing a period's `endTime`** doesn't reschedule already-armed expiry jobs — and `schedulePassExpiry`'s own comment says reconcile cannot correct a stale fire time on a live delayed job (`apps/passes-api/src/lib/queue.ts:26-30`).

Minimum fix: block destination delete while non-terminal passes reference it (409, matching the scheduleType pattern) and filter `deletedAt: null` in the promotion query. The occupancy/period-edit staleness is acceptable if reconcile runs frequently — but write that assumption down.

## Minor

- **M2-adjacent asymmetry:** GET/PATCH `/api/schools/:id` are SUPER_ADMIN-only (`apps/schools-api/src/routes/school.ts:62`, `:82`) while every sub-resource is school-scoped — a school ADMIN cannot read their own school's name/timezone. The frontend will need this; add school-scoped read.
- **Duplicate enum sources:** `PassStatus`/`Role` exist both as Prisma enums and as const objects in `@hallpass/types`; `apps/passes-api/src/schemas/passes.ts:2` imports from `@hallpass/db` while routes import from `@hallpass/types`. They agree today; nothing enforces it. Pick one import source per layer (types package for API surface) or add a compile-time assertion.
- **Bulk user failures are opaque:** every rejection maps to `"Failed to create user"` (`apps/user-api/src/routes/user.ts:188`) — a duplicate email is indistinguishable from an FK error, which matters for CSV onboarding retries.
- **`GET /passes` orders by `id asc`** (`apps/passes-api/src/routes/passes.ts:321`) — a live pass board wants newest/in-flight first; retrofitting sort order onto cursor pagination later is awkward, so decide now.
- **Quota check is read-then-write** (`apps/passes-api/src/routes/passes.ts:176-199`): concurrent create/complete cycles can exceed `maxPerInterval` by one. The partial unique index bounds the damage; acceptable, but note it.
- **Socket auth swallows DB errors as "Unauthorized"** (`apps/passes-api/src/lib/socket.ts:42-44`) while the HTTP path deliberately surfaces them as 500 (the `sessionUser.ts` contract) — same inconsistency the recent auth fix removed from HTTP.
- **Status-code conventions drift:** school-not-found is 404 in policy PUT (`apps/schools-api/src/routes/policy.ts:64-68`) but 422 in pass create (`apps/passes-api/src/routes/passes.ts:97-100`); `requireSchool` uses 422 where 403 is arguable. Worth one convention note in docs.
- **`demo-ui`** (`apps/demo-ui/app.js`) is plain JS with no tests/lint in a TS monorepo — fine as a demo, but mark it explicitly non-production.
- Open items 1, 2, 4, 5, 6 in `tech-debt.md` (midnight edge-case family, slot-key TTL over-admit, calendar-date helper, Redis factory, PORT schema drift) remain valid; not repeated here.

## High-level design assessment

**What's working well, keep it:**
- Consistent layering in all three apps (`app.ts` → middleware → zod schema → route handler → Prisma) with `*_SELECT` constants and `toXResponse` mappers pinning the wire contract to `@hallpass/types` (the `PassRow`/`toPassResponse` compile-time trick at `apps/passes-api/src/routes/passes.ts:61-67` is a nice pattern — worth copying to the other apps).
- Concurrency discipline in passes-api: every state transition is a status-guarded `updateMany` with a 409 on lost races; slot accounting is atomic Lua with in-script rollback; expiry jobs are idempotent with stable job ids; reconcile is the designated recovery path. This is the strongest code in the repo.
- Shared-database, three-service split is pragmatic at this scale. Don't split the database; do write down table ownership (users → user-api, school config → schools-api, passes → passes-api) so cross-service reads (passes-api reads `School`, `Period`, `SchoolCalendar`, `Destination`, `PassPolicy`, `User`) stay read-only by convention.

**Drift to arrest:**
- The three apps are copy-paste siblings already diverging: `requireSelfOrRole` exists only in user-api, `requireMinRole` only in passes-api, passes-api's `resolveSessionUser` refactor never made it back to the other two, PORT env schemas differ. This is the middleware-package problem (below).
- `POST /passes` is a ~210-line handler with 8 sequential DB round trips (`apps/passes-api/src/routes/passes.ts:82-294`). It's correct, but it's the file every future business rule lands in. When it next grows, extract a `lib/passService.ts` (period resolution / quota check / create-student-pass / create-teacher-pass) — not a full service layer everywhere, just this hot spot. The CRUD handlers in schools-api/user-api do **not** need a service layer.
- `ROLE_RANK` placing SERVICE (4) above SUPER_ADMIN (3) is a load-bearing decision duplicated in three files with no comment. `requireMinRole(TEACHER)` admits SERVICE everywhere. Fine if intended — say so once, in the shared package.

## Recommendation: `@hallpass/express-middleware` package

Confirmed — this is the right next move (tech-debt item 3), and the audit found the concrete contents. Proposed shape:

```
packages/middleware/            → "@hallpass/express-middleware"
  src/
    auth.ts          createRequireAuth(auth) + resolveSessionUser(auth, headers)
                     — port the passes-api version (it has the correct
                       error semantics); parameterize by auth instance
    roleGuard.ts     roleRank, requireRole, requireMinRole, requireSelfOrRole
                     — the superset of all three copies, with the SERVICE-rank
                       decision documented once
    schoolScope.ts   requireSchoolAccess (schools-api) + requireSchool (passes-api)
    validate.ts      validateBody / validateQuery / validateParams (identical ×3 today)
    rateLimit.ts     shared limiter factories — fix C2 here once (per-user keying,
                     optional Redis store)
    health.ts        createHealthRoute(serviceName) (identical ×3 today)
    errorHandler.ts  notFound + createErrorHandler(logger) (identical ×3 today)
    cors.ts          parseCorsOrigins(env) (passes-api's lib/cors.ts, generalized)
    env.ts           baseEnvSchema (DATABASE_URL, BETTER_AUTH_*, CORS_ORIGIN,
                     PORT as z.coerce.number()) — apps .extend() it; fixes item 6
    express.d.ts     the Request.user augmentation (identical ×3 today)
```

**Adoption order (each step is independently shippable):**
1. `validate.ts`, `health.ts`, `errorHandler.ts`, `express.d.ts` — byte-identical today, zero-risk extraction.
2. `roleGuard.ts` — merge the superset; no behavior change per app.
3. `auth.ts` — port passes-api's `resolveSessionUser` into `createRequireAuth(auth)`; user-api and schools-api pick up the better error contract (their current copies 401 on `getSession` throw but 500 on DB error — semantics already match).
4. `rateLimit.ts` + `env.ts` + `cors.ts` — this is where the C2 fix lands.

Keep `roleGuard`'s pure functions unit-tested in the package; the per-app middleware tests (schools-api has the fullest set under `apps/schools-api/tests/middleware/`) move with the code.
